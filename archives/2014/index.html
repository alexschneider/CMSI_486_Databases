F1 Database

F1 database is a distributed database developed by Google - more details here: http://static.googleusercontent.com/media/research.google.com/en/us/pubs/archive/41344.pdf

I’m going to do my best to explain my understanding of what the paper describes.

The premise of what Google was trying to do was create a RELATIONAL database designed to be distributed and highly available, unlike other relational databases such as MySQL which didn’t have quite as great of a distributed system. It was designed to be Scalable, consistent, and usable as a SQL database.

Some of the design choices that Google had made increased reading and writing latency due to extra safety and security of data, though application level programming reduced the hit to the user experience.

  • Servers are stateless - clients can connect to any server in the network and expect the same result
  • Queries can be executed parallelized and map-reduced, and will transparently be determined if that will be beneficial.
  • Primary keys are designed such that queries can easily be run in parallel
  • Downtime is to be minimized
  • Multiple types of transactions to gaurentee consistency with data retrieved and committed.
  • Locking types are flexible
  • Logging is designed to be efficient and descriptive
  • Document based store is available along with relational stores

Pros and Cons of Mongo Db

In addition to my coursework in MongoDB, I decided to do some more research from someone who may not be as biased as 10gen (the developers of MongoDB).

Here are the two links I used:

http://halls-of-valhalla.org/beta/articles/the-pros-and-cons-of-mongodb,45/

http://www.sarahmei.com/blog/2013/11/11/why-you-should-never-use-mongodb/

(Now, granted, they may be biased against MongoDB, obviously the 2nd one, but we’ll see).

R.E. the first one:

  • Advantages:
    • Load balancing and redundancies in MongoDB is far simpler than many SQL databases.
    • The document database system (instead of relational database system) means that the speed of MongoDB is far faster than anything a relational database system can achieve, if only because one query is often sufficient to obtain all or nearly all information.
    • MongoDB is flexible
  • Disadvantages:
    • Joins must be handled in the program, which can reduce code readability. Also, traditional optimizations that databases can make for joins no longer exist in Mongo
    • MongoDB requires much more memory than relational databases
    • Denormalization is common in MongoDB. To normalize, sometimes storing references is required (in which case, why not just use a relational database in the first place?)
    • Though MongoDB’s actions are mostly atomic, it doesn’t have the same concept of ACID transactions that many relational databases have.
    • MongoDB is up and coming technology, as opposed to relational databases which are battle hardened.
    • Safe mode in mongodb is confusing - data sent to the database doesn’t ensure that data gets written to teh database
    • Transactions need to be manually emulated in code.

The first article claims that even though the cons list is long, there are some use cases for it.

Second article:

The author is a self-proclaimed expert on web applications who feels he made the wrong decision with MongoDB.

The project was a widely distributed social networking system that involved a lot of independent applications working together to provide a social networking experience. Many databases were asynchronously updating and had different contents, and would only fetch the ones they needed.

Mongo documents are great for self contained documents that have many separate components, but for social networking, some of the components end up being duplicated quite often. Social networking isn’t really self-contained either, which makes linking between them difficult.

Part of the issue the author ran into was that cache invalidation is rather difficult to implement when considering edge cases. MongoDB acts more like a cache than a database.

As a result, MongoDB’s use case is extremely narrow, according to the author, perhaps narrow enough to warrant ignoring or avoiding it alltogether.

Methods and Procedures

Unfortunately, I cannot post the research paper that I’ve been involved with here, but here is a bit of a writeup about it:

There had already been a database system set up prior to me joining the project, so I was tasked with maintaining and preparing the database for publication, along with analytics and other minor things. We did run into some problems, namely that even though the original designers of the database schema attempted to anticipate the needs of the students for creating proper logs, due to the nature of how medical diagnoses and procedures go, students were forced to adapt in unfortunately inconsistant ways. A non-insignificant amount of the data would be skewed against certain types of diagnoses and procedures unless something was done.

My involvement was on the programatic side of things, to develop a solution to perform automated cleanup (i.e. typos or improper categorization) and facilitate manual cleanup when automatic cleanup was impossible (procedures/diagnoses missing/not present). One major problem that was run into was consistently deciding what diagnoses related to the often vague symptoms. This required us to improve the context available to the person making the judgments beyond what was immediately available, requiring complex joins, groups, and sometimes, queries from separate databases altogether.

The research is nearly done, but it tested the limits of what I knew about database administration and programming, and forced me to think in new and creative ways to implement what the others on the project required.

Chapter 18 Problems

Notes:

  • Index access helps to reduce the amount of time required to search through a file - stores each value and a pointer to all disk blocks that include it.
  • Primary index used to physically order files on the disk
  • Clustering index - used if multiple indexes are suitable for physical ordering
  • Secondary - doesn’t deal with ordering of the file, but locates it.
  • Dense indexes have an index entry for every single search key
  • Sparse - only some of the search keys
  • Indexes are useful because they occupy a smaller portion of the disk than the data file - they include fewer records and they only have two fields, the key and the address of the block.
  • Clustering - takes up more room, but works on more complex files and indexes
  • Secondary index - allows other means of accessing data when a primary index already exists - must still be unique.
    • Takes more storage space and a longer search time, but still improvement over no index at all.
    • For non-unqiue data, indexing is still possible, but makes searches more complicated, since the index has to store a list of pointers, instead of a single one.
  • Multilevel indexes
    • Used to reduce the search space for a binary search
    • Uses multiple levels of indexes until the highest level can fit in a single block.
    • Can be used for any index, so long as there are distinct values for K(i)
  • Partitioned hashing - n components produce n hashes, which are catted together to produce one hash. Useful for equality, but not really for searching
  • B-tree and B+-tree
    • Tree is formed of nodes, each node (except for the top node) has one parent and 0 or more children.
    • Trees are unbalanced
    • B-Trees
      • Kept between 50-100% full
      • Searching works by forcing things to the left of the key to be left than it, and things to the right of the key to be greater than it. That way, when you compare the key you are looking for to the tree, you can easily find out which direction to travel.
      • Balancing a tree minimizes the number of levels
      • B-trees have additional constraints to make sure that they are always balanced
      • Each node of a b-tree has multiple bits of data and multiple tree pointers - the heirarchy of increasing value goes left to right - tree pointer, data pointer, tree pointer, data pointer, etc, etc, and finally a tree pointer.
      • If removal causes the node to be less than half full, it’s combined with neighboring nodes
    • B+-trees
      • Like a b-tree, except each node only points to the node to the left most node at the next level, and the left node points to the node to the right, and so on
      • Insertions and deletions call for complicated algorithms, but they are computationally fast.
  • Grid files - multi-dimensional array. Works well for access time, but requires a lot of space and difficult to re-organize
  • Hash index - uses hashing other than the one used for primary index.
  • Bitmap indexes - index for one or more columns, and each value in the column is indexed.
    • Built using one particular value and is an array of bits (sequential 1s and 0s referenceing whether or not something exists at that location).
    • Efficient in storage space, but difficult to maintain
    • Can use bitwise operations to maintain though.
  • Function based indexing - applying a function on field(s) becomes the key to an index.
  • Logical vs Physical indexes
    • Physical indexes have a block location and an offset that must be changed if the file location on the disk is changed
    • Logical indexes - let some other system handle where the files are, and use indexes to point to it.

Chapter 17 Problems

Notes:

  • Most databases are stored somewhere in the file system (on magnetic or SSD storage).
  • They generally have algorithms to help accessibility, called indexes
  • Databases are too big to fit entirely into memory
  • Magnetic tape storage allows for a very cheap method of backing up data
  • File allocation is done in clusters, which are contigious bits of data that are linked (provide pointers to the next node). Indexes are also common.
  • File headers provide description about a file that tells the OS how to process it
  • File operations include retrieval and update operations - finding information or adding new information to a file.
    • Searches can only be done by pattern matching, so complex queries need to be simplified to be able to work on the filesystem.
  • Some files have ordered records, while others are nothing more than ‘heaps’. Inserting new records is efficient, but searching/location less so.
  • Sorted records take more time for inserts, but searching is much faster. Binary searches can be done easily (but require some sort of object that is sortable).
  • Hashing allows fast access under certain search conditions - a hash can yield the address where the file is stored - files are placed in buckets
  • RAID - use many disks to provide speed or redundency increases

17.37. Can you think of techniques other than an unordered overflow file that can be used to make insertions in an ordered file more efficient?

Insertions can be made more efficient by keeping multiple ordered files, appending or prepending where appropriate, or if a record needs to be in the middle of the file, split it - rejoin it when there is downtime.

  • Sigma is used to denote SELECT (Selecting certain rows in a table and discarding others)
  • R refers to an expression whose result is a relation (like a name of a database relation)
  • Boolean operations work the way one may expect…
  • Pi is used to denote PROJECT (selecting some columns from the table and discarding others). Basically the way it works in MongoDB. SELECT statements in SQL perform both selection and projection
  • Pi_(R)
  • RENAME lets you change the names to make them more descriptive (uses rho)
  • Unions, intersections, and set differences work basically the way one may expect them to
  • Cartesian products
    • Cross product/cross join - denoted by X
    • Produces new element by combining every member from one relation with another (and doing every possible permutation).

    • Equijoins (=) have one or more pair of attributes that have identicial values in every tuple
    • Natural joins (*) are equijoins that get rid of the 2nd set of attributes that represent a join
  • The prior operations is a complete set - any other operation can be expressed as a series of the ones so far listed
  • Division operations are applied to two relations where one’s attributes are a subset of another - get rid of a field that may include duplicate data that is irrelevant

Chapter 5 Problems

Notes:

  • SQL also has some advanced features beyond basic CRUD, like TRIGGERs and ASSERTIONs
  • SQL treats NULLs in a specific way regarding booleans - statements only return TRUE/FALSE if they are short-circuitable with the comparsion
  • SQL allows us to check for nulls by using ‘IS’ (like python!)
  • SQL allows sets and nested queries
    • Keywords like ANY or ALL can be useful for nested queries and using comparison operators
    • Nested query scope is similar to programming languages - the scope refers to the innermost nested query
  • Exists - check to see if something is there
    • Can be used with “EXCEPT” as well to provide conditions
  • Distinct - for explicit sets
  • AS for renaming stuff
  • Joins…
    • Default is inner
    • left and right are cool too
  • Aggregation - similar to mongodb, grouping, count, sum, max, min, avg, etc.
    • Having - groups with conditions
  • Triggers and assertions:
    • Assertions for enforcing constraints (and preventing updates when something doesn’t meet them)
    • Triggers for creating actions when certain events occur, such as updates or deletes
  • Views
    • Single table derived from other tables that can have queries executed on. Will always remain up to date.
    • View updating is complicated. It can generally be passed through, but not in certain conditions. Usually views are not updatable.
  • Schema changes
    • Dropping - can be done in a cascading or restricted fashion (cascading means dropping all references, restrict prevents dropping if any references exist)
    • Alter - add or remove columns, or change column properties
